# Deep Agents 进阶：上下文管理与记忆

深度 Agent 的一大特点是能在 **长时间运行** 中保持有效工作。但 LLM 的上下文窗口是有限的——随着对话轮次增加，Agent 很容易"忘记"之前做过的事或"迷失"在细节中。Deep Agents 通过多种机制解决这个问题。

## 上下文窗口的挑战

一个典型的深度 Agent 执行过程：

```
第 1 轮：用户请求 → Agent 制定计划                    [已用 2K tokens]
第 2 轮：Agent 搜索文件 → 返回大量内容                [已用 8K tokens]
第 3 轮：Agent 读取文件 → 更多内容                    [已用 15K tokens]
第 4 轮：Agent 调用工具 → 工具返回长输出              [已用 25K tokens]
...
第 N 轮：接近上下文窗口上限                          [已用 120K tokens]
         → Agent 开始遗忘早期信息，行为退化
```

Deep Agents 采用三种策略应对：

| 策略 | 机制 | 效果 |
|------|------|------|
| **自动摘要** | 压缩旧的对话历史 | 减少 token 占用 |
| **虚拟文件系统** | 将中间结果写入文件 | 按需读取，避免全部加载 |
| **Todo 规划** | 维护任务清单 | 保持目标聚焦 |

## 自动摘要（Auto-Summarization）

当对话历史变得过长时，Deep Agents 会自动触发摘要机制：

```
对话历史（原始）：
┌────────────────────────────────────────┐
│ [Message 1] 用户：分析这个项目          │
│ [Message 2] Agent：好的，让我先查看...  │
│ [Message 3] Tool：文件列表（500行）     │  ← 压缩为摘要
│ [Message 4] Agent：我发现了3个模块...   │
│ [Message 5] Tool：模块A的代码（300行）  │  ← 压缩为摘要
│ [Message 6] Agent：模块A负责...         │
├────────────────────────────────────────┤
│ [Message 7] Agent：现在分析模块B...     │  ← 保留完整内容
│ [Message 8] Tool：模块B的代码           │
│ [Message 9] Agent：分析完成，开始写报告 │
└────────────────────────────────────────┘

对话历史（摘要后）：
┌────────────────────────────────────────┐
│ [Summary] 用户请求分析项目，Agent 已完  │
│ 成模块A的分析，发现3个模块...          │  ← 浓缩的摘要
├────────────────────────────────────────┤
│ [Message 7] Agent：现在分析模块B...     │  ← 保留完整内容
│ [Message 8] Tool：模块B的代码           │
│ [Message 9] Agent：分析完成，开始写报告 │
└────────────────────────────────────────┘
```

这个过程对用户完全透明，Agent 会自动在需要时触发。

## 虚拟文件系统

Deep Agents 的文件系统工具（`read_file`、`write_file`、`edit_file`）操作的是一个 **虚拟文件系统**，存储在 LangGraph 的 state 中：

```python
from deepagents import create_deep_agent
from langchain.chat_models import init_chat_model
from dotenv import load_dotenv

load_dotenv()

agent = create_deep_agent(
    model=init_chat_model("openai:gpt-4o"),
)

result = agent.invoke({
    "messages": [{
        "role": "user",
        "content": "研究 Python 的 asyncio 库，把笔记写到 notes.md 中"
    }]
})
```

Agent 执行时会这样使用文件系统：

```
Agent 思考：我需要系统研究 asyncio，先记笔记。

→ write_file("notes.md", "# Asyncio 研究笔记\n\n## 核心概念\n...")
→ write_file("notes.md", 追加 "## 事件循环\n...")  
→ read_file("notes.md")  // 回顾之前的笔记
→ write_file("notes.md", 追加 "## 协程与任务\n...")
→ write_file("report.md", 基于笔记撰写最终报告)
```

### 文件系统的三大用途

**1. 存储中间结果**

```python
# Agent 可能的行为：
# 步骤 1：研究并保存发现
write_file("findings/module_a.md", "模块A的分析结果...")
write_file("findings/module_b.md", "模块B的分析结果...")

# 步骤 2：后续需要时按需读取
content = read_file("findings/module_a.md")
```

**2. 作为子代理间的共享工作区**

```python
# 主 Agent 写入任务描述
write_file("task_brief.md", "请分析以下数据集...")

# 子代理读取任务并写入结果
# task(researcher): read_file("task_brief.md") → write_file("analysis.md", ...)

# 主 Agent 读取子代理的结果
read_file("analysis.md")
```

**3. 保存大输出避免占用上下文**

当工具返回特别长的输出时，Deep Agents 会自动将输出保存到文件中，而不是直接放入对话历史：

```
工具返回 10,000 行代码 → 自动保存到 output_12345.txt
对话中只记录："输出已保存到 output_12345.txt（10,000 行）"
Agent 需要时调用 read_file("output_12345.txt") 按需读取
```

## Todo 规划作为上下文工程

`write_todos` 工具表面上是一个任务管理工具，但它更深层的作用是 **上下文工程**：

```python
# Agent 使用 write_todos 的典型流程：

# 开始时：制定计划
write_todos([
    {"task": "分析项目结构", "status": "in_progress"},
    {"task": "检查依赖关系", "status": "pending"},
    {"task": "评估代码质量", "status": "pending"},
    {"task": "撰写分析报告", "status": "pending"},
])

# 完成一项后：更新状态
write_todos([
    {"task": "分析项目结构", "status": "done"},
    {"task": "检查依赖关系", "status": "in_progress"},  # 更新
    {"task": "评估代码质量", "status": "pending"},
    {"task": "撰写分析报告", "status": "pending"},
])
```

::: tip 为什么说 Todo 是"上下文工程"？
从技术角度看，`write_todos` 几乎是一个 **no-op**（空操作）——它不执行任何实际逻辑。但它的价值在于：每次调用时，Agent 需要重新审视当前进度和下一步计划，这相当于在对话中插入了一个"路标"，帮助 Agent 在长时间运行中保持方向感。
:::

## 长时间运行的策略

### 策略一：分块处理

对于需要处理大量数据的任务，将工作分成小块逐步完成：

```python
from deepagents import create_deep_agent
from langchain_core.tools import tool
from langchain.chat_models import init_chat_model
from dotenv import load_dotenv

load_dotenv()

@tool
def read_csv_chunk(file_path: str, start: int, size: int) -> str:
    """分块读取 CSV 文件"""
    import csv
    rows = []
    with open(file_path) as f:
        reader = csv.reader(f)
        for i, row in enumerate(reader):
            if i < start:
                continue
            if i >= start + size:
                break
            rows.append(",".join(row))
    return "\n".join(rows)

agent = create_deep_agent(
    model=init_chat_model("openai:gpt-4o"),
    tools=[read_csv_chunk],
    system_prompt="""处理大文件时，请遵循以下策略：
1. 使用 read_csv_chunk 分块读取数据（每次 100 行）
2. 每处理一块，将结果写入虚拟文件系统
3. 使用 write_todos 追踪已处理的进度
4. 全部处理完成后，读取所有中间结果撰写最终报告
""",
)

result = agent.invoke({
    "messages": [{
        "role": "user",
        "content": "分析 data.csv 中的销售数据，生成月度汇总报告"
    }]
})
```

### 策略二：检查点恢复

利用 LangGraph 的 checkpointer 实现可恢复的长任务：

```python
from deepagents import create_deep_agent
from langgraph.checkpoint.memory import MemorySaver
from langchain.chat_models import init_chat_model
from dotenv import load_dotenv

load_dotenv()

checkpointer = MemorySaver()

agent = create_deep_agent(
    model=init_chat_model("openai:gpt-4o"),
    checkpointer=checkpointer,
)

config = {"configurable": {"thread_id": "analysis-task-001"}}

# 第一次运行
result = agent.invoke(
    {"messages": [{"role": "user", "content": "开始分析项目"}]},
    config=config,
)

# 如果中断了，可以从检查点恢复
result = agent.invoke(
    {"messages": [{"role": "user", "content": "继续之前的分析"}]},
    config=config,  # 相同的 thread_id 会恢复之前的状态
)
```

### 策略三：子代理分治

将大任务拆解给多个子代理，每个子代理只处理一部分，避免单个 Agent 上下文溢出：

```
主 Agent 计划：
├── 子代理 A：处理第 1-1000 行    → 写入 result_a.md
├── 子代理 B：处理第 1001-2000 行 → 写入 result_b.md
├── 子代理 C：处理第 2001-3000 行 → 写入 result_c.md
└── 主 Agent：读取三个结果文件 → 生成汇总报告
```

## 核心概念回顾

| 概念 | 说明 |
|------|------|
| 自动摘要 | 自动压缩旧的对话历史，减少 token 占用 |
| 虚拟文件系统 | 基于 LangGraph state 的文件存储，用于保存中间结果 |
| 大输出存文件 | 工具长输出自动保存为文件，按需读取 |
| `write_todos` | Todo 规划工具，兼具上下文工程的作用 |
| 分块处理 | 将大任务拆分为小块逐步完成 |
| 检查点恢复 | 利用 checkpointer 实现可恢复的长任务 |

## 小结

本篇我们学习了：

1. 长时间运行的 Agent 面临上下文窗口溢出的核心挑战

3. 虚拟文件系统的三大用途：中间结果、共享工作区、大输出存储
4. `write_todos` 作为上下文工程手段帮助 Agent 保持方向感
5. 三种长任务策略：分块处理、检查点恢复、子代理分治
