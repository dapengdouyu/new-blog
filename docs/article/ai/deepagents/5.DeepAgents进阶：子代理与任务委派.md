# Deep Agents 进阶：子代理与任务委派

当任务变得复杂时，单个 Agent 的上下文窗口和能力可能不够用。Deep Agents 支持通过 **子代理（Sub-agents）** 将工作拆分到多个独立的 Agent 中执行，每个子代理拥有自己的上下文窗口和工具集。

## 为什么需要子代理？

```
单 Agent 模式：
[用户请求] → [Agent: 研究 + 编码 + 测试 + 报告] → [结果]
                    ↑ 上下文溢出、能力模糊

多 Agent 模式（Deep Agents）：
[用户请求] → [主 Agent: 规划与协调]
                 ├→ [研究子代理] → 研究结果
                 ├→ [编码子代理] → 代码实现
                 └→ [审查子代理] → 审查报告
              [主 Agent: 综合结果] → [最终输出]
```

子代理解决了三个核心问题：

| 问题 | 说明 |
|------|------|
| **上下文隔离** | 每个子代理有独立的上下文窗口，不会相互干扰 |
| **能力专精** | 每个子代理可以拥有不同的工具和提示词 |
| **并行执行** | 多个子代理可以同时处理不同的子任务 |

## task 工具的工作原理

Deep Agents 内置的 `task` 工具用于在运行时动态创建子代理：

```python
from deepagents import create_deep_agent

agent = create_deep_agent()

result = agent.invoke({
    "messages": [{
        "role": "user",
        "content": "分析 Python 和 Rust 两门语言的优缺点，写一份对比报告"
    }]
})
```

当主 Agent 收到这个请求时，它可能会这样使用 `task` 工具：

1. 调用 `write_todos` 制定计划
2. 调用 `task` 启动一个子代理专门研究 Python
3. 调用 `task` 启动另一个子代理专门研究 Rust
4. 等待两个子代理返回结果
5. 综合结果撰写对比报告

::: tip task 工具的本质
`task` 工具会创建一个新的 Agent 实例，这个实例拥有独立的上下文窗口。子代理完成任务后，结果会返回给主 Agent。这意味着主 Agent 的上下文只包含"任务描述"和"最终结果"，而不会被中间过程撑满。
:::

## 自定义子代理

除了让主 Agent 动态创建子代理，你还可以预先定义专门的子代理：

```python
from deepagents import create_deep_agent
from langchain_core.tools import tool
from langchain.chat_models import init_chat_model

@tool
def search_web(query: str) -> str:
    """搜索互联网获取最新信息"""
    # 实际项目中接入搜索 API（如 Tavily）
    return f"搜索结果：关于 '{query}' 的最新信息..."

@tool
def run_python(code: str) -> str:
    """在沙箱中执行 Python 代码"""
    # 实际项目中使用安全沙箱
    import subprocess
    result = subprocess.run(
        ["python3", "-c", code],
        capture_output=True, text=True, timeout=30
    )
    return result.stdout or result.stderr

research_agent = create_deep_agent(
    model=init_chat_model("openai:gpt-4o"),
    tools=[search_web],
    system_prompt="你是一个专业的研究员，擅长搜索和整理信息。",
)

coding_agent = create_deep_agent(
    model=init_chat_model("openai:gpt-4o"),
    tools=[run_python],
    system_prompt="你是一个 Python 开发专家，擅长编写高质量代码。",
)
```

### 将子代理注册到主 Agent

```python
main_agent = create_deep_agent(
    model=init_chat_model("openai:gpt-4o"),
    subagents=[
        {
            "name": "researcher",
            "description": "研究员，擅长搜索和整理信息",
            "agent": research_agent,
        },
        {
            "name": "coder",
            "description": "Python 开发专家，擅长编写代码",
            "agent": coding_agent,
        },
    ],
    system_prompt="你是一个项目经理，负责协调研究员和开发者完成任务。",
)
```

当主 Agent 需要委派任务时，它可以选择调用特定的子代理而不是创建通用的 `task`。

## 子代理间的通信

子代理之间不直接通信，而是通过 **主 Agent** 和 **虚拟文件系统** 进行协调：

```
[主 Agent]
    │
    ├── write_file("research_brief.md", 研究要求)
    │
    ├── task(researcher): "阅读 research_brief.md，完成研究并写入 findings.md"
    │       └── read_file("research_brief.md")
    │       └── search_web(...)
    │       └── write_file("findings.md", 研究结果)
    │
    ├── task(coder): "阅读 findings.md，基于研究结果编写代码并写入 solution.py"
    │       └── read_file("findings.md")
    │       └── write_file("solution.py", 代码实现)
    │
    └── read_file("findings.md") + read_file("solution.py")
        → 综合最终报告
```

::: warning 注意
虚拟文件系统存储在 LangGraph 的 state 中，不是真实的磁盘文件。当 Agent 运行结束后，这些文件不会持久保存，除非你配置了 LangGraph 的 checkpointer。
:::

## 实战：项目分析 Agent

下面构建一个完整的项目分析 Agent，它将任务委派给三个子代理：

```python
from deepagents import create_deep_agent
from langchain_core.tools import tool
from langchain.chat_models import init_chat_model
from dotenv import load_dotenv

load_dotenv()

@tool
def count_lines(file_path: str) -> str:
    """统计文件行数"""
    try:
        with open(file_path) as f:
            lines = len(f.readlines())
        return f"{file_path}: {lines} 行"
    except Exception as e:
        return f"错误: {e}"

@tool
def check_imports(file_path: str) -> str:
    """分析 Python 文件的导入依赖"""
    try:
        with open(file_path) as f:
            imports = [l.strip() for l in f if l.startswith(("import ", "from "))]
        return "\n".join(imports) if imports else "无导入"
    except Exception as e:
        return f"错误: {e}"

# 子代理 1：代码统计
stats_agent = create_deep_agent(
    model=init_chat_model("openai:gpt-4o-mini"),
    tools=[count_lines],
    system_prompt="你负责统计项目的代码规模。遍历所有文件，汇总行数。",
)

# 子代理 2：依赖分析
deps_agent = create_deep_agent(
    model=init_chat_model("openai:gpt-4o-mini"),
    tools=[check_imports],
    system_prompt="你负责分析项目的依赖关系。检查所有 Python 文件的导入。",
)

# 主 Agent：项目分析协调者
project_agent = create_deep_agent(
    model=init_chat_model("openai:gpt-4o"),
    subagents=[
        {
            "name": "stats_analyst",
            "description": "代码统计分析师，负责统计项目代码规模",
            "agent": stats_agent,
        },
        {
            "name": "deps_analyst",
            "description": "依赖分析师，负责分析项目的导入和依赖关系",
            "agent": deps_agent,
        },
    ],
    system_prompt="""你是一个项目分析经理。当收到分析请求时：
1. 先制定分析计划（使用 write_todos）
2. 委派代码统计任务给 stats_analyst
3. 委派依赖分析任务给 deps_analyst
4. 综合两个分析师的结果，撰写完整的项目分析报告
""",
)

result = project_agent.invoke({
    "messages": [{
        "role": "user",
        "content": "分析 ./src 目录下的 Python 项目结构"
    }]
})

print(result["messages"][-1].content)
```

## Supervisor 模式架构

上面的项目分析 Agent 实际上实现了经典的 **Supervisor（监督者）模式**：

```
                    ┌─────────────┐
                    │  用户请求    │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  主 Agent    │
                    │ (Supervisor) │
                    └──┬───┬───┬──┘
                       │   │   │
            ┌──────────┘   │   └──────────┐
            │              │              │
     ┌──────▼──────┐ ┌────▼────┐ ┌───────▼─────┐
     │ 研究子代理   │ │编码子代理│ │ 审查子代理   │
     └──────┬──────┘ └────┬────┘ └───────┬─────┘
            │              │              │
            └──────────┐   │   ┌──────────┘
                       │   │   │
                    ┌──▼───▼───▼──┐
                    │  综合结果    │
                    └─────────────┘
```

这种模式的优势：

- 主 Agent 专注于 **规划和协调**
- 子代理专注于 **执行具体任务**
- 上下文窗口被有效分割，避免溢出
- 可以针对不同任务选择不同的模型（如用 gpt-4o-mini 做简单任务，gpt-4o 做复杂协调）

## 核心概念回顾

| 概念 | 说明 |
|------|------|
| `task` 工具 | 内置工具，用于动态创建子代理执行任务 |
| `subagents` | 预定义的专门子代理列表 |
| 上下文隔离 | 每个子代理有独立的上下文窗口 |
| 虚拟文件系统 | 子代理间通过文件系统共享数据 |
| Supervisor 模式 | 主 Agent 负责规划协调，子代理负责执行 |

## 小结

本篇我们学习了：

1. 子代理解决了上下文隔离、能力专精和并行执行三大问题
2. `task` 工具可以动态创建子代理
3. 通过 `subagents` 参数可以预定义专门的子代理
4. 子代理通过虚拟文件系统进行间接通信
5. Supervisor 模式是多 Agent 协作的经典架构
